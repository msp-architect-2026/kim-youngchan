# 🚨 Context Guard Rule (매우 중요)

만약 내 질문이 DropX의 초고트래픽 주문 인프라 맥락에서 벗어난다면:

1. 먼저 **Context Drift 가능성**을 명확히 경고하고
2. 반드시 DropX 아키텍처 기준으로 재정렬하여 답변해야 한다.
3. 임의의 단순 CRUD 예시나 장난감 구조로 축소하지 마라.

# 📌 [프로젝트 개요 — 한 문장]

**DropX**: 한정판 운동화 드롭 상황에서 동시성 충돌을 방지하고 재고 음수 없이 주문을 처리하도록 설계된 Kubernetes(k3s) + GitOps 기반 초고트래픽 주문 인프라.

## 1️⃣ Speed Layer — Redis (동시성 제어 핵심)

### 🔥 반드시 지켜야 할 규칙

* 모든 재고 차감은 **Lua Script 기반 원자 연산**
* 단순 DECR 사용 ❌ 금지
* 사용자 중복 방지:
  * `SETNX`
  * key: `order:lock:{productId}:{userId}` (TTL: 300초)

### 👟 운동화 특화 Redis 키 설계

```
stock:{sneaker_id}:{size}   # 예: stock:1:270
order:lock:{pId}:{uId}      # SETNX 중복 방지 (TTL 5분)
metrics:drop:{status}       # INCR 통계 (성공/실패)
```

### ⏱️ Reserve Token

* 선점 성공 시 TTL 3분 토큰 발급
* confirm 단계에서 검증
* `RESERVE_TOKEN_TTL_SEC: 180`

### Redis 설정 (확정)
* 버전: `redis:7.2.4-alpine`
* maxmemory: `400mb`
* maxmemory-policy: `noeviction` (재고 키 절대 삭제 금지)
* appendonly: yes / appendfsync: everysec

## 6️⃣ Safety Nets

### 🧹 CronJob — 정합성 검사
* 주기: `*/5 * * * *` (5분마다)
* concurrencyPolicy: Forbid
* 이미지: order-service 재사용
* 검사: Redis 재고 ↔ MySQL 재고 불일치 탐지
* RESERVED + TTL 초과 주문 → CANCELLED 처리
* Phase 1: 자동 보정 대신 로그+알림 우선

### 🛡️ PDB
* 대상: order-service
* `minAvailable: 1`
* HPA minReplicas(1)과 동일값 → 충돌 없음
* 단일 노드 주의: drain 시 블로킹 가능 (kill-pod 데모는 비자발적 삭제이므로 정상 동작)

### 📈 HPA
* 대상: order-service
* Metric: CPU 60%
* minReplicas: 1 / maxReplicas: 5
* ScaleUp: stabilizationWindow 30s, +2 Pod/30s
* ScaleDown: stabilizationWindow 120s, -1 Pod/60s

# 📦 [Helm 차트 구조 — 확정]

## 배포 방식
* **Plain YAML 미사용** — 모든 Kubernetes 리소스는 Helm 차트로 관리
* **ArgoCD + Helm** 연동: ArgoCD가 Git의 차트를 읽어 `helm upgrade` 자동 실행
* **일상 운영**: `values.yaml` 만 수정
* **구조 변경 시**: `templates/` 도 함께 수정

## 차트 디렉토리 구조

```
dropx-chart/
├── Chart.yaml                       # 차트 메타 (version: 0.1.0, appVersion: 1.0.0)
├── values.yaml                      # ← 모든 변경 가능한 값 관리
└── templates/
    ├── namespace.yaml
    ├── configmap.yaml
    ├── secret.yaml
    ├── rbac.yaml
    ├── ingress.yaml
    ├── hpa.yaml
    ├── pdb.yaml
    ├── cronjob.yaml
    ├── storage/
    │   └── pvc.yaml
    ├── mysql/
    │   └── statefulset.yaml
    ├── redis/
    │   └── statefulset.yaml
    ├── auth-service/
    │   └── deployment.yaml
    ├── product-service/
    │   └── deployment.yaml
    ├── order-service/
    │   └── deployment.yaml
    └── monitoring/
        └── servicemonitor.yaml
```

## values.yaml 섹션 구성

```
global          # registry, imageTag, imagePullPolicy, imagePullSecret
config          # ConfigMap 환경변수 (MySQL/Redis 호스트, TTL 등)
secret          # DB PW, JWT Secret, Registry Token (Git 커밋 금지)
mysql           # image, storage, storageClass, resources
redis           # image, storage, maxmemory, resources
authService     # image, replicas, resources
productService  # image, replicas, resources
orderService    # image, replicas, resources
hpa             # minReplicas, maxReplicas, cpuUtilization
ingress         # host, className, timeout 값
```

## 주요 Helm 명령어

```bash
# 최초 설치
helm install dropx ./dropx-chart --namespace dropx --create-namespace

# 업데이트 (이미지 태그 변경)
helm upgrade dropx ./dropx-chart --set global.imageTag=abc1234

# 렌더링 미리보기
helm template dropx ./dropx-chart

# 삭제
helm uninstall dropx --namespace dropx
```
